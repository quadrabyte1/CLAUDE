- ☐

  Assume there is a safety requirement that Action A must complete every time in no more than 1000 msec

  - ☐

    We need a requirement for this for sure

- ☐

  Assume Action A is composed of sub-actions: X, Y, Z

  - ☐

    At the moment:

    - ☐

      X takes 300 msec

    - ☐

      Y takes 300 msec

    - ☐

      Z takes 300 msec

- ☐

  Everything is working fine

- ☐

  I maintain we should not have requirements for the durations of X, Y, and Z

  - ☐

    We shouldn't care how long the three sub actions take as long as the Action A can still meet its requirement

  - ☐

    If the implementation changed such that X now takes 600 msec but Y only takes 3, we shouldn't have to change any requirements because X, Y, and Z times were implementation details and are allowed to be fungible

- ☐

  I think you were concerned that there might be a change that caused X, Y, or Z to change times but, without a requirement (and a guaranteed test to be sure it is followed), we can't be sure the 1000 msec will still be met and that is a valid concern:

  - ☐

    It may still be quite important (for some reason) that the times of X, Y, and Z be within certain budgets but that doesn't mean there should be a REQUIREMENT to establish those budgets--if the system will meet the Action A requirement when the necessary verification test(s) run then we can be sure that safety, at least with respect to this requirement, is assured

  - ☐

    As a matter of good design practices we should probably have a Design Document that lays out the X, Y, and Z actions along with a specification, perhaps, for each action's execution time budget. And we might want a requirement to state "...shall perform Action A in accordance with document R0164548 Design Document: Critical Action A..."

  - ☐

    When verification gets to this latest requirement, they need only go to the developers and ask "can you send me the proof that you tested for adherence to R0164548?" but the testing burden is passed to the developers

  - ☐

    Lastly, if there was a requirement that X had to perform within some specific interval, the verification team would be tasked with coming up with a test case that shows compliance. but that may be really tough to determine if X is deep in the bowels of the Action A-responsible Software Item. It should be far easier for the developers to come up with code/hooks/etc. to satisfy the engineering requirement that the specification is met. Note also, that in a case like this where Verification must reach deeply into a functional block to prove a requirement is satisfied, our present policies allow for a "testability change" to the code to provide a way to get the necessary information--I am strongly against this practice in principle. Ttesting a modified bit of code only proves the modified code works as desired but says approximately nothing about whether the production (unmodified) code will also work as desired.
